## 문제

`shouldForNone`이 `shouldForAll`에 비해 극심한 성능 저하를 보임.

```kotlin
// 느림: >1초
elements.shouldForNone { it.shouldBe(target) }

// 빠름: ~100ms (10배 차이)
elements.shouldForAll { it.shouldNotBe(target) }
```

이슈 신고자 환경에서 동일한 논리를 표현하는데 대폭 성능 차이가 발생. 재현 환경(컬렉션 크기, JVM 워밍업, matcher 복잡도)에 따라 실제 차이는 달라질 수 있음.

## 근본 원인 (예상)

`shouldForNone`의 구현이 **모든 요소에 대해 실패 정보를 수집**하는 구조일 가능성이 높음.

- `shouldForNone`: "모든 요소가 매칭에 실패해야 함" → 각 요소의 실패 정보를 모아서 보고
- `shouldForAll`: "모든 요소가 매칭에 성공해야 함" → 첫 실패 시 즉시 중단

`shouldForNone`에서는 모든 요소가 실패하는 것이 **정상 동작**인데, 불필요하게 에러 메시지를 생성하고 수집하는 오버헤드가 발생하는 것으로 추정.

## 해결 방향

### 접근법 1: 에러 수집 최적화

`shouldForNone` 내부에서 매칭 성공 여부만 판단하고, 실패(=정상) 케이스에서는 에러 메시지를 생성하지 않도록 최적화.

### 접근법 2: 조기 종료

`shouldForNone`에서 하나라도 매칭에 성공하면 즉시 실패 → 에러 메시지는 성공한 요소에 대해서만 생성.

### 분석 필요 사항

1. `shouldForNone` 내부 구현 확인 (에러 수집 로직)
2. `shouldForAll` 내부 구현과 비교
3. GH-5245 관련 이슈에서 논의된 내용 확인
4. 벤치마크 코드 작성으로 개선 폭 측정

| **종합 머지 확률** | **중상** (성능 개선 자체는 환영하지만, 실패 리포팅 품질과의 트레이드오프 논의 가능) |

## 포트폴리오 가치

**매우 높음** — 이 이슈가 가장 높은 포트폴리오 가치를 가진 이유:

1. **성능 최적화는 백엔드 엔지니어 핵심 역량** — 면접에서 "오픈소스 테스팅 프레임워크의 성능 병목을 분석하고 개선했다"고 말할 수 있음
2. **내부 아키텍처 이해 필요** — 단순 수정이 아니라 어설션 평가 엔진의 동작 원리를 파악해야 함
3. **측정 가능한 결과** — before/after 벤치마크로 명확하게 개선 폭을 보여줄 수 있음
4. **KSentinel 테스트에서 실제로 사용** — 체감 성능 향상 스토리

## 실행 계획

1. `shouldForNone` / `shouldForAll` 내부 구현 코드 분석
2. 재현 벤치마크 작성 (before 측정)
3. 에러 수집 로직 최적화 또는 조기 종료 구현
4. 벤치마크 재실행 (after 측정)
5. 이슈에 분석 결과 + 벤치마크 첨부하며 코멘트
6. PR 제출 (벤치마크 결과 포함)

## 리스크

- **실패 리포팅 메시지 품질 보존** — 메인테이너가 "성능은 좋지만 에러 메시지가 덜 유용해졌다"고 판단하면 리젝 가능. 최적화 시 실패 메시지 품질을 유지하는 설계가 필수
- 에러 수집 로직이 다른 곳에서도 사용되면 사이드 이펙트 가능
- 성능 개선이 특정 케이스에서만 유효할 수 있음 → 다양한 시나리오 벤치마크 (컬렉션 크기, matcher 복잡도 등)
- `apiDump` 변경이 필요할 수 있음 (내부 API 변경 시)
